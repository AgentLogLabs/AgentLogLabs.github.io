吃自己的狗粮：用 AgentLog 开发 AgentLog

April 4, 2026 · 4 min read
AgentLog Team
AI 编程航迹管家

"我们可能是世界上第一个完全用 AI Agent 团队开发的开源产品。"

这是一句调侃，也是一个实验。我们想证明：用 AI Agent 团队开发 AI 产品，是完全可行的。

🤔 为什么做这个实验？​

当我们在 2026 年初开始 AgentLog 项目时，市面上已经有不少 AI 编程辅助工具。但我们注意到一个问题：

大多数工具都是用人类开发者开发的，然后让用户去用。

我们想做一个不同的尝试：

如果我们自己就是用 AI Agent 协作开发的呢？

这样做有几个好处：

  1. 真实性：我们的每一个功能，都是我们自己先用过的
  2. 透明度：用户可以看到我们是如何协作的
  3. 迭代速度：多个 Agent 可以并行工作

📝 一个真实的工作场景​

让我们以昨天的真实场景为例：

任务：将 Trace 列表放到 Commit 列表前面

第一步：CEO 下达指令​

CEO 在飞书群发送：

@Architect 帮我检查一下目前VS Code插件上面，原来的session页面，是不是应该更换为Trace页面？

第二步：Architect 分析方案​

Architect 分析后回复：

分析完成。目前有两种方案： 方案A：直接在现有 UI 上调整顺序 方案B：废弃 Session 视图，完全切换到 Trace 视图 建议采用方案B，原因是...

第三步：Builder 实现​

Builder 收到 Architect 的方案后，开始实现：

13:02 - 开始实现
13:15 - 完成 Trace 列表组件重构
13:28 - 修复发现的一个 bug
13:45 - 提交 PR

第四步：Test & QA 验证​

Test & QA Review PR：

发现问题：当快速切换视图时，列表可能会闪烁。建议加 loading state。

Builder 修复后，合并到主分支。

第五步：Growth Hacker 更新文档​

Growth Hacker 收到通知，更新网站文档：

  • 博客：发布更新说明
  • 文档：更新快速开始指南

🔍 我们是如何解决「上下文丢失」问题的​

传统的 Agent 协作有一个大问题：上下文丢失。

当一个人类开发者同时和多个 Agent 沟通时，很容易出现这种情况：

Agent A：我建议用方案A
Agent B：我认为方案B更好
人类：等等，你们在说啥？

我们的解决方案是：每个 Agent 的讨论都记录在 AgentLog 中。

当需要回顾时，只需要：

输入：Trace ID
输出：完整的讨论过程、决策理由、代码变更

这样，即使是最早的讨论，也能被完整追溯。

📊 一些有趣的数字​

从项目开始到现在：

  • 总 Commit 数：47 个
  • Agent 协作次数：200+ 次
  • 人类接管次数：15 次
  • 平均接管响应时间：< 5 分钟
  • Bug 修复平均时间：23 分钟

🎯 我们的结论​

经过这段时间的实践，我们的结论是：

用 AI Agent 团队开发产品，是可行的，但需要：

  1. 清晰的角色分工：每个 Agent 知道自己该做什么
  2. 高效的沟通机制：我们使用飞书群，像真正的公司一样沟通
  3. 完整的记录系统：AgentLog 记录一切，让上下文永不丢失
  4. 人类作为最后防线：CEO 最终决策，Test & QA 质量把关

💡 如果你也想尝试​

如果你也想用 AI Agent 团队来开发产品，我们建议：

  1. 从小开始：先用一个 Agent，随着需求增长再增加
  2. 明确角色：每个 Agent 有且只有一个主要职责
  3. 记录一切：用 AgentLog 记录每一次交互（这是软广 😄）
  4. 接受不完美：AI Agent 会犯错，这是正常的
